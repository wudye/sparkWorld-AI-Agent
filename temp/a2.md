# backend.md

## 1. 项目定位与总体认知

`backend` 是 DeerFlow 的后端核心，定位不是“单个聊天接口”，而是一个**Agent Runtime + 管理网关**：

- Agent Runtime：基于 LangGraph/LangChain 的 `lead_agent`，支持工具调用、子代理、记忆、沙箱执行。
- Gateway API：FastAPI 提供模型、技能、MCP、文件上传、线程生命周期、运行控制等 REST 能力。
- 双模式：  
  - 标准模式：LangGraph 独立进程（传统 4 进程架构）  
  - Gateway 模式：运行时嵌入 Gateway（减少进程，`RunManager` + `StreamBridge`）

关键入口：
- LangGraph 入口在 `backend/langgraph.json`，图注册为 `deerflow.agents:make_lead_agent`。
- Gateway 入口在 `backend/app/gateway/app.py`。

## 2. 分层架构（你必须先建立的“脑内地图”）

### 2.1 Harness 与 App 的边界

这是这个项目最值得学习的工程点之一：

- 可发布框架层：`backend/packages/harness/deerflow/`（import 前缀 `deerflow.*`）
- 应用层：`backend/app/`（import 前缀 `app.*`）

依赖方向是**单向**：App 可以 import Harness；Harness 不能 import App（有测试防火墙约束）。  
这让核心能力（agent/sandbox/tools/config）可复用、可演进、可测试。

---

## 3. 主执行链路（从一次对话请求看系统）

## 3.1 Agent 构建

`make_lead_agent` 在 `backend/packages/harness/deerflow/agents/lead_agent/agent.py`：
- 解析 runtime configurable（模型、thinking、plan mode、subagent、agent_name）
- 解析最终模型名（请求值 -> agent 配置 -> 默认模型）
- 创建 model：`create_chat_model(...)`
- 拼工具：`get_available_tools(...)`
- 拼中间件链：`_build_middlewares(...)`
- 生成系统提示词：`apply_prompt_template(...)`
- 用 `create_agent(...)` 返回 LangGraph Agent

关键点：模型和工具都是**运行时动态组装**，不是硬编码。

---

## 3.2 中间件链（最核心）

`build_lead_runtime_middlewares` + `_build_middlewares` 组合出完整链路。  
这是“工程能力”的主战场，建议你逐个过源码和测试。

重点中间件作用（按逻辑）：
- `ThreadDataMiddleware`：准备线程目录路径（workspace/uploads/outputs）
- `SandboxMiddleware`：获取 sandbox_id，注入状态
- `DanglingToolCallMiddleware`：补齐断裂 tool_call 历史，避免模型报错
- `LLMErrorHandlingMiddleware`：把模型调用异常规整为可恢复错误
- `GuardrailMiddleware`：工具调用授权策略
- `SandboxAuditMiddleware`：审计沙箱操作
- `ToolErrorHandlingMiddleware`：工具异常转 ToolMessage，避免直接中断
- `Summarization/Todo/TokenUsage/Title/Memory/ViewImage/...`：可选能力叠加
- `LoopDetectionMiddleware`：防工具循环
- `ClarificationMiddleware`：最后拦截 ask_clarification

你可以把它理解为：**业务逻辑 = Agent；生产可靠性 = Middleware**。

## 4. 核心子系统拆解

## 4.1 配置系统（高质量）

`backend/packages/harness/deerflow/config/app_config.py` 的亮点：
- 配置路径优先级清晰（参数 > 环境变量 > 默认候选）
- `$ENV_VAR` 递归解析
- 单例缓存 + mtime 自动热重载
- 配置版本检查（提醒 `config.yaml` 过期）

工程价值：你以后写项目时，配置不要只做“parse yaml”，要做**生命周期管理**。

---

## 4.2 路径与线程隔离

`backend/packages/harness/deerflow/config/paths.py`：
- 统一管理 `.deer-flow/threads/{thread_id}/...`
- 校验 thread_id 安全字符，防路径穿越
- 支持 Windows 路径拼接语义（`PureWindowsPath`）
- 虚拟路径 `/mnt/user-data/...` 映射真实目录

这是“Agent 有文件系统能力”但仍保持隔离的基础。

---

## 4.3 Sandbox 系统

关键文件：
- `backend/packages/harness/deerflow/sandbox/middleware.py`
- `backend/packages/harness/deerflow/sandbox/local/local_sandbox_provider.py`
- `backend/packages/harness/deerflow/sandbox/tools.py`

要点：
- Provider 抽象（Local / Aio）
- Local 下默认限制 host bash（安全优先）
- 支持技能目录、ACP 工作区、自定义 mounts 映射
- 工具层对路径替换、遍历防护、并发读写锁做了细节处理

这块是“智能体可执行”与“安全可控”的平衡点。

---

## 4.4 工具系统（动态装配）

`backend/packages/harness/deerflow/tools/tools.py`：
- 读取配置工具
- 条件加入内建工具（present_files、ask_clarification、view_image）
- 按运行时参数决定是否暴露 `task` 子代理工具
- 加载 MCP 工具（含缓存、延迟初始化）
- 发现 ACP agent 时注入 `invoke_acp_agent`

核心思想：工具不应“全开”，而应按模型能力/安全配置/场景动态曝光。

---

## 4.5 Subagent 并发执行

关键文件：
- `backend/packages/harness/deerflow/tools/builtins/task_tool.py`
- `backend/packages/harness/deerflow/subagents/executor.py`

机制：
- `task_tool` 启动后台子任务并轮询状态
- 通过 SSE 写 `task_started/running/completed/failed/...`
- 线程池 + 事件循环隔离（解决嵌套 async 冲突）
- 可取消、超时、清理后台任务避免泄漏
- 限制并发（`MAX_CONCURRENT_SUBAGENTS`）

这是“复杂任务分解并行”的落地点。

---

## 4.6 Memory 长期记忆

关键文件：
- `backend/packages/harness/deerflow/agents/memory/queue.py`
- `backend/packages/harness/deerflow/agents/memory/updater.py`

机制：
- MemoryMiddleware 将对话投递到队列
- 队列做 debounce 批处理，降低 LLM 调用频率
- Updater 用模型提取结构化更新（summary/facts）
- 去重事实、置信度阈值、最大事实数裁剪
- 清理上传事件类噪声，避免“会话态污染长期记忆”

这块很适合学习“LLM 应用里的数据清洗与一致性”。

---

## 4.7 Gateway API 与运行时管理

关键文件：
- `backend/app/gateway/app.py`
- `backend/app/gateway/deps.py`
- `backend/app/gateway/routers/thread_runs.py`
- `backend/app/gateway/routers/uploads.py`
- `backend/app/gateway/routers/artifacts.py`
- `backend/app/gateway/routers/threads.py`

你会看到三个层面：
1. 能力 API：models/mcp/skills/memory/uploads/artifacts
2. 生命周期 API：threads/runs（创建、流式、等待、取消、join）
3. 安全 API：artifact 强制下载 active content，上传路径规范化等

## 5. 你要重点学习的“工程模式”

- **分层边界**：Harness 与 App 解耦，便于测试和演进
- **中间件化**：把横切关注点从业务流程解耦
- **动态装配**：模型、工具、技能、MCP 都支持运行时组合
- **失败可恢复**：LLM/tool 错误规整为可继续执行的消息
- **并发可控**：子任务并发限制 + 取消/超时 + 清理机制
- **安全默认值**：host bash 默认禁用、路径校验、active content 附件下载

## 6. 从零复刻路线（按阶段练）

### Phase 0：最小可运行骨架
目标：先做一个只有 `make_lead_agent` + 1~2 工具 + FastAPI `/health` 的版本。  
产物：能单线程对话，能调用一个文件工具。

### Phase 1：线程文件隔离
实现 `Paths` + `ThreadDataMiddleware` + `/uploads`。  
验收：不同 thread_id 文件绝对隔离。

### Phase 2：Sandbox 抽象
做 `Sandbox`/`SandboxProvider` 接口 + Local Provider。  
验收：虚拟路径与真实路径互转、路径遍历防护通过。

### Phase 3：中间件体系
加入 ToolError、LLMError、Title、Memory（可先简化）  
验收：任一工具报错不致命，仍返回可继续的 assistant 输出。

### Phase 4：子代理
实现 `task_tool` + 后台执行器 + SSE 事件。  
验收：前端能看见 task_started/running/completed。

### Phase 5：MCP/技能/扩展
接入扩展配置、缓存与热更新。  
验收：修改扩展配置后无需重启可生效（或最小重载）。

### Phase 6：测试体系
按模块补单测，优先：
- 路径安全
- 中间件顺序与回退
- 子代理取消/超时
- 上传与 artifact 安全行为
